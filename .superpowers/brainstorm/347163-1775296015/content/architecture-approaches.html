Architecture Approach

How should we structure the main process and domain logic?

A

Layered Clean Architecture

Main process layers: Domain → Use Cases → Adapters → IPC

Pure domain models (Worktree, Session, Repo) with no dependencies. Use cases orchestrate operations (CreateWorktree, CleanStaleWorktrees). Adapters wrap git CLI, tmux, filesystem. IPC handlers are thin — validate input, call use case, return result.

Pros

  • Domain logic is testable without git/tmux
  • Adapters are swappable (mock for tests)
  • Use cases map 1:1 to IPC handlers
  • Scales well as features grow

Cons

  • More files and indirection upfront
  • ~15 files for main process alone
B

Service Layer (Pragmatic)

Main process layers: Services → IPC

Group logic into service modules (GitService, TmuxService, WorktreeService, ThemeService). Services call git/tmux directly. IPC handlers call services. No separate domain models — services own their types.

Pros

  • Fewer files, less indirection
  • Still well-organized and testable
  • Natural mapping to what exists

Cons

  • Services tend to grow fat over time
  • Git/tmux coupling harder to swap out
  • Complex operations span multiple services
C

Hybrid: Services + Ports

Main process layers: Services → Ports (interfaces) → Adapters → IPC

Services contain business logic but depend on port interfaces (GitPort, TmuxPort, FileSystemPort) rather than calling CLI directly. Adapters implement the ports. Best of both worlds — organized like B but with the testability of A.

Pros

  • Clean dependency inversion without full DDD ceremony
  • Services stay focused, adapters stay swappable
  • ~10-12 main process files — moderate
  • Ports document the system boundaries explicitly

Cons

  • Port interfaces need upfront design
  • Slightly more abstract than B

Recommendation: C (Hybrid)

It gives you clean architecture benefits (testability, swappable adapters) without the ceremony of full DDD. The port interfaces also serve as documentation — anyone reading GitPort immediately understands what git operations the app needs. And since you want to share this app, having adapters means others could potentially swap tmux for a different terminal multiplexer.
